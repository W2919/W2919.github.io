二、总结思考

2.2.1 设想和目标

  1. 我们的软件要解决智能驾驶场景下的安全辅助问题，核心聚焦司机状态监测、道路车辆检测、驾驶相关便捷功能集成，定义清晰。典型用户为私家车主、网约车司机等，典型场景包括长途驾驶时的司机疲劳/危险行为提醒、行驶过程中道路车辆实时预警、驾驶过程中的导航与多媒体娱乐协同。
  2. 原计划实现的8大核心功能已完成7项，分别是：登录注册（密码+手机短信）、主界面框架与实时信息显示（时间、天气）、视频采集与存储、照片拍摄与管理、道路车辆检测、导航系统、音乐播放模块、用户信息管理；仅司机状态检测的语音提醒功能未完全实现。项目按原计划交付时间完成中期开发，因处于中期阶段，暂未面向外部用户推广，未涉及用户量目标达成。
  3. 目前仅团队内部测试使用，核心功能中道路检测、视频/照片存储的易用性符合预期，导航系统的路径规划准确性达到预想，但稳定性有待提升，整体离“智能驾驶安全辅助”的最终目标更近一步。
  4. 经验教训：前期对导航系统的兼容性测试不足，导致出现程序奔溃问题；模型训练数据集的多样性可进一步提升。若重新来过，会在项目初期增加模块间兼容性测试计划，同时扩大数据集覆盖场景（如不同天气、路况下的车辆图像）。

2.2.2 计划

  1. 项目初期预留了1周的计划时间，时间充足，已完成需求分析、技术选型、任务拆分与分工。
  2. 计划阶段组员意见分歧主要集中在模型选型（YOLOv11 vs YOLOv8），通过对比两者训练速度、检测精度及适配性，结合项目硬件环境，最终投票选择YOLOv11。
  3. 原计划的核心功能基本完成，未完成的“司机状态语音提醒”因模型训练后的接口适配延迟，暂未实现。
  4. 暂无发现无价值的工作，所有已完成模块均为项目核心流程或用户高频使用功能。
  5. 每项任务均有明确交付件，例如“登录功能”交付可运行的登录界面、后端验证逻辑与数据库交互代码；“视频存储模块”交付视频加载、播放、删除的完整功能与界面。
  6. 项目整体基本按计划进行，意外情况为导航系统在高并发数据请求时出现程序奔溃，未提前预估到高德地图API的响应延迟与本地数据处理的冲突。
  7. 计划中预留了3天缓冲区，主要用于处理模块集成问题，已有效解决了视频流与模型检测的性能冲突问题。
  8. 将来计划会增加“模块兼容性测试时间”，每个功能模块开发完成后，需与已完成模块进行联调测试；同时将缓冲区调整为5天，应对突发的技术问题。
  9. 学到了“任务拆分需细化到最小可执行单元”，避免因任务边界模糊导致开发延迟。若重来，会在计划阶段制定更详细的联调测试计划，明确每个模块的接口标准。

2.2.3 资源

  1. 硬件资源（Windows 10操作系统、高性能GPU电脑）、软件资源（PyCharm、Qt Designer、MySQL）、开发库（PyQt5、OpenCV、YOLOv11）均充足，能满足开发需求。
  2. 任务时间估计基于成员过往开发经验，精度较高，例如“主界面开发”预估5天，实际完成时间4.5天；“模型训练”预估7天，实际因数据集优化延长至9天。
  3. 测试资源不足，仅依赖团队内部手动测试，无专门测试工具与测试人员；美工设计资源未低估，界面设计通过Qt Designer完成，风格统一，未出现难度超预期情况。
  4. 部分重复性工作（如视频/照片列表的加载逻辑）可由同一成员统一开发后复用，避免重复编码，提升效率。
  5. 经验教训：未提前规划测试资源，导致部分隐藏Bug未及时发现。若重来，会引入简单的自动化测试工具（如pytest），同时明确1名成员兼职负责模块测试。

2.2.4 变更管理

  1. 变更消息通过团队微信群与GitHub Issues同步，确保每个成员及时知晓，例如“模型训练数据集调整”“导航系统API更换”等变更均第一时间同步。
  2. 采用“核心功能优先”原则决定功能优先级，例如“司机状态检测的基础框选”列为必须实现，“语音提醒”因复杂度较高推迟至后期；非核心的“歌词同步显示”功能暂不实现。
  3. 项目中期出口条件定义清晰：核心功能可正常运行、界面交互逻辑流畅、无致命Bug（如程序奔溃、数据丢失）。
  4. 针对可能的变更（如API接口调整），提前预留了备用方案，例如导航系统除高德地图API外，预留了百度地图API的适配接口。
  5. 组员能有效处理意外工作请求，例如临时增加“视频倍速播放”功能，负责该模块的曾煜林利用缓冲区时间完成开发。
  6. 学到了变更管理需“及时同步+明确优先级”，避免因变更导致开发方向混乱。若重来，会制定正式的变更申请流程，通过文档记录变更原因、影响范围与实施计划。

2.2.5 设计/实现

  1. 设计工作在项目初期（第1周）由队长吴昊牵头，结合组员技术特长完成架构设计，各模块详细设计由对应负责组员完成，时间与人员分配合适。
  2. 设计阶段的模糊点为“视频流与模型检测的同步逻辑”，通过召开小组技术研讨会，参考YOLO官方文档与开源项目案例，最终确定“多线程分离采集与检测”的方案。
  3. 团队未使用单元测试与TDD，仅通过手动测试验证功能；使用Git进行版本控制，Qt Designer用于UI设计，这些工具有效提升了开发效率与协作流畅度。
  4. 项目初期未制定详细UML文档，仅绘制了简单的架构流程图，目前各模块的实际实现与初期架构一致，后续会补充UML类图与时序图。
  5. 导航系统产生的Bug最多，原因是高德地图API的响应数据格式与本地数据处理逻辑冲突，且未考虑网络延迟情况；发布后发现的重要Bug为“连续切换导航路线时程序奔溃”，开发时未模拟高并发场景下的路线切换。
  6. 代码复审通过GitHub Pull Request进行，组员提交代码后需由至少1名其他组员审核通过方可合并，严格执行代码规范（如变量命名、注释格式）。
  7. 学到了“架构设计需充分考虑模块间接口兼容性”，避免后期集成冲突。若重来，会在设计阶段增加接口文档定义，同时引入单元测试覆盖核心逻辑。

2.2.6 测试/发布

  1. 团队暂无正式测试计划，仅在每个模块开发完成后进行手动测试（功能是否可用、界面是否正常）；未进行正式验收测试，中期阶段以内部联调测试为主。
  2. 未使用专门的测试工具，仅依赖Qt的调试功能与print日志输出定位问题。
  3. 通过手动操作验证功能效能，例如视频播放的流畅度、模型检测的响应速度，测试工作有效发现了部分功能Bug，但遗漏了高并发场景下的问题；后续需引入自动化测试工具，增加场景化测试（如不同网络环境、硬件配置下的运行情况）。
  4. 发布（内部测试版本）过程中发现的意外问题：部分音频格式（如flac）不支持、照片列表加载速度慢，已通过扩展音频解码库、优化图片加载逻辑解决。
  5. 学到了测试是保障项目质量的关键，中期阶段的测试不足会导致后期问题集中爆发。若重来，会制定分阶段测试计划：模块测试→集成测试→场景测试，每个阶段明确测试重点与负责人。
